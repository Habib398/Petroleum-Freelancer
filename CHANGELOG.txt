FASE 1 — REFACTORIZACION DE ASSETS FRONTEND
================================================================================

OBJETIVO
--------
Extraer todo CSS y JavaScript inline de los archivos HTML a archivos
estaticos externos, mejorando el cacheo, mantenibilidad y rendimiento.

PATRON IMPLEMENTADO
-------------------
- CSS con variables dinamicas de marca (Jinja2):
  Se retuvo un bloque <style> minimo con :root { --var: {{ valor }} }
  El resto del CSS se movio al archivo .css externo.

- JS con datos del servidor:
  Se implemento el patron "Configuration Bridge":
  Un objeto global window.NombreConfig = { dato: "{{ valor }}" }
  El archivo .js externo lee ese objeto al iniciarse.

MIGRACION DEL MODULO STATIONS (Blueprint nativo)
-------------------------------------------------
El modulo stations fue el primero en migrarse completamente a un
Blueprint real de Flask:

  ANTES (routes/stations.py):
    def register(app):
        stations_bp = Blueprint("stations", __name__)  # dentro de register
        ctx = app.extensions['ctx']
        login_required = ctx.login_required            # cerrado en variable
        @stations_bp.get("/api/stations")
        def api_stations(): ...
        app.register_blueprint(stations_bp)

  AHORA (modules/stations/routes.py):
    from routes.auth import login_required, role_required  # importados
    bp = Blueprint("stations", __name__)                   # nivel de modulo

    @bp.get("/api/stations")
    @login_required
    def api_stations():
        ctx = current_app.extensions['ctx']   # accedido dentro de la funcion
        ...

  Diferencias clave:
  - Blueprint a nivel de modulo (no encapsulado en una funcion)
  - Decoradores importados directamente, no cerrados sobre ctx
  - ctx accedido via current_app.extensions['ctx'] en cada view
  - Patron estandar y oficial de Flask Blueprints
