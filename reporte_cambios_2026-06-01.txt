1. MÓDULO DE INCIDENCIAS — NUEVO FLUJO
---------------------------------------

Se rediseñó por completo el funcionamiento del módulo de
incidencias según lo acordado con el cliente.

Quién puede hacer qué:
  - Operador: registra incidencias de su estación.
  - Jefe de estación: registra y, sobre todo, marca el estado
    de cada incidencia. Es su obligación atenderlas.
  - Administrador: ya NO ve la bitácora viva. Solo accede al
    reporte mensual (ver punto 3).

Nuevos estados de una incidencia:
  - Pendiente: recién registrada, el jefe aún no la revisa.
  - Leída: el jefe la reconoció pero todavía no actúa.
  - Atendida: el jefe la resolvió en sitio.
  - Reportada: el jefe la escaló (queda registro de quién y
    cuándo).

Cuando una incidencia pasa de "Pendiente" a cualquier otro
estado, el sistema graba automáticamente quién fue el jefe
que la atendió y la fecha exacta. Esto se conserva aunque
después cambie de estado, para mantener la trazabilidad.


2. NOTIFICACIONES AUTOMÁTICAS
------------------------------

  - Cuando el operador registra una incidencia nueva, el jefe
    de estación recibe una notificación inmediata.
  - El menú lateral del jefe muestra un contador rojo con la
    cantidad de incidencias pendientes por revisar. Se
    actualiza solo cada 30 segundos.
  - Si entra a la bitácora, ve un banner arriba con el número
    de pendientes para que no pueda pasarlas por alto.


3. REPORTE MENSUAL DEL ADMINISTRADOR
-------------------------------------

Como el admin ya no ve las incidencias del día a día, se creó
una pantalla nueva para que pueda supervisar:

  - Acceso desde: menú lateral "Supervisión → Reporte de
    incidencias", o desde el menú de módulos del admin.
  - Selecciona una estación, año y mes, y se muestran:
      * 4 tarjetas grandes con totales por estado (Pendientes,
        Leídas, Atendidas, Reportadas) con colores tonales.
      * Total del mes y tiempo promedio que tarda el jefe en
        leer cada incidencia.
      * Desglose por severidad (Crítica / Alta / Media / Baja).
      * Tabla detallada con cada incidencia: fecha, folio,
        severidad, estado, título, descripción, quién la creó,
        quién la leyó y quién la cerró.
  - Botón "Descargar PDF" genera el reporte mensual completo
    listo para imprimir o archivar.


4. REDISEÑO VISUAL DE LA BITÁCORA
----------------------------------

La pantalla de incidencias del operador y del jefe quedó más
formal y más fácil de leer:

  - Las incidencias ya no se muestran en cuadrícula de dos
    columnas con tarjetas grandes. Ahora son una lista
    vertical de filas compactas, una debajo de otra.
  - Filtros como pestañas tipo "chip" arriba (Todas /
    Pendientes / Leídas / Atendidas / Reportadas), cada una
    con su contador en vivo.
  - El formulario para registrar incidencias quedó oculto por
    defecto. Se abre al presionar el botón "+ Nueva
    incidencia" y se cierra al guardar.
  - Cada fila muestra de un vistazo: severidad (con color e
    ícono), folio, estado (con badge de color), título,
    descripción resumida y la metadata (estación, fecha, quién
    la creó, quién la leyó).
  - Las descripciones largas se cortan a dos renglones con un
    enlace "Ver más" para no saturar la pantalla.
  - Las incidencias atendidas y reportadas se muestran con
    transparencia para que destaquen visualmente las que aún
    requieren atención.
  - Los botones del jefe aparecen según el estado de cada
    incidencia: "Marcar leída" (azul), "Atender" (verde),
    "Reportar" (morado).


5. LIMPIEZA DEL DASHBOARD EJECUTIVO
------------------------------------

Como el admin ya no maneja incidencias en vivo, se quitaron
de su dashboard ejecutivo:

  - Las gráficas que sumaban incidencias junto con alertas y
    tareas ahora solo cuentan alertas y tareas.
  - El conteo por estación ya no incluye incidencias.
  - El link viejo "Incidencias" del menú del admin fue
    reemplazado por "Reporte de incidencias", que lleva a la
    pantalla del punto 3.

Para el jefe y el operador el link sigue apuntando a la
bitácora normal.


6. MIGRACIÓN DE DATOS EXISTENTES
---------------------------------

Las incidencias que ya existían en el sistema (con los
estados anteriores: Abierta / En progreso / Cerrada) fueron
migradas automáticamente al nuevo modelo:

  - Abierta       → Pendiente
  - En progreso  → Leída
  - Cerrada      → Atendida

Se conservaron las fechas originales, los responsables y la
trazabilidad. Ningún dato se perdió.


7. PUNTOS PENDIENTES (acordados con el cliente)
------------------------------------------------

Ninguno por ahora. El módulo quedó al 100% con lo solicitado.

A futuro pueden considerarse mejoras opcionales:
  - Incluir fotos como evidencia al registrar una incidencia.
  - Notificar al operador cuando el jefe atienda o reporte su
    incidencia.
  - Reporte mensual consolidado (todas las estaciones en un
    solo PDF) para el admin.

REPORTE DE CAMBIOS — MÓDULO DE ACTIVIDADES (MODELO CLASSROOM)
Fecha: 1 de junio de 2026
================================================================


RESUMEN
-------

Se rediseñó por completo el módulo de actividades para que
funcione como un sistema de tipo "classroom": el administrador
o jefe asigna actividades, el operador las completa subiendo
evidencia, y los supervisores revisan los resultados.

No hubo cambios en la base de datos estructuralmente; se
aprovecharon las columnas existentes y se remapearon estados
históricos de forma automática.


1. NUEVO MODELO DE PERMISOS
----------------------------

  Administrador:
    - Crea, edita y elimina cualquier actividad.
    - Ve todas las evidencias de todas las estaciones.
    - Accede al Calendario Operativo (no al de actividades).

  Jefe de estación:
    - Crea actividades solo para su estación.
    - Puede editar/eliminar las que él creó, no las del admin.
    - Ve evidencias de su estación.
    - Accede al Calendario Operativo.

  Operador:
    - Solo ve su calendario de actividades.
    - Completa actividades subiendo evidencia (foto o PDF).
    - No puede ver ni descargar evidencias de otros.
    - Una vez completada, no puede resubir (como en classroom).


2. CAMBIOS EN EL FLUJO
-----------------------

  ANTES:
    Operador sube evidencia → Jefe/Admin aprueba o rechaza.

  AHORA:
    Operador sube evidencia → Actividad queda COMPLETADA.
    No hay paso de aprobación.

  El estado "rechazada" se conserva solo para casos donde el
  admin necesite anular manualmente una entrega incorrecta.


3. PANTALLAS Y NAVEGACIÓN
--------------------------

  Nueva pantalla:
    - "Evidencias por estación" (/mod/station-evidence):
      Solo para jefe y admin. Muestra tabla de evidencias con
      filtros por estación, fecha y búsqueda. Permite descargar
      los archivos adjuntos.

  Cambios de acceso:
    - /mod/activities → Solo operadores. Admin/jefe son
      redirigidos automáticamente al Calendario Operativo.
    - /mod/operational-calendar → Solo admin y jefe de estación.
      Incluye botón "Crear actividad" y enlace a evidencias.

  Menú lateral:
    - Operadores ven: "Actividades".
    - Admin/Jefe ven: "Calendario operativo" y
      "Evidencias por estación".


4. CRITERIO DE ACTIVIDAD COMPLETADA
-------------------------------------

  En todo el sistema (reportes, dashboard, KPIs, recordatorios)
  se unificó el criterio:

    Completada = status <> 'rejected'

  Antes se usaba status = 'approved', lo que dejaba fuera
  entregas válidas con status 'submitted' o 'reviewed'.


5. MIGRACIÓN DE DATOS HISTÓRICOS
----------------------------------

  Al arrancar el sistema se aplicó una migración automática:

    approved  → submitted
    reviewed  → submitted

  Esto garantiza que las entregas históricas se cuenten como
  completadas bajo el nuevo criterio. Sin pérdida de datos.


6. CORRECCIONES TÉCNICAS RESUELTAS
------------------------------------

  - Calendario operativo no actualizaba al cambiar de mes:
    Se corrigió sincronizando las fechas visibles con el
    callback datesSet de FullCalendar.

  - Botón "Crear actividad" no respondía:
    La variable de sesión de usuario no estaba disponible
    al momento de registrar el listener del botón.

  - Actividades completadas no se reflejaban en verde:
    El API no retornaba el campo "completed" al calendario
    operativo. Se corrigió en el backend.

  - Evento del primer día del mes desaparecía:
    SQLite comparaba strings de fecha con timestamps ISO
    (ej. "2026-06-01" vs "2026-06-01T00:00:00"), lo que
    excluía siempre el primer día del rango visible.
    Se corrigió cortando el parámetro a solo la fecha.

  - Formulario de entrega visible tras completar actividad:
    El operador podía volver a subir evidencia aunque ya
    hubiera completado. Se bloqueó correctamente.

  - Admin y jefe no veían evidencias al abrir una actividad:
    El API siempre devolvía submission=null para el admin.
    Se corrigió para que ambos roles vean la entrega registrada.


7. LIMPIEZA DE BASE DE DATOS (ambiente de pruebas)
----------------------------------------------------

  Se eliminaron registros de prueba acumulados:

    calendar_events       9,752 registros eliminados
    agenda_calendar_events   25 registros eliminados
    submissions             706 registros eliminados

  Las plantillas de actividades (154) se conservaron intactas.
  Existe respaldo de la base de datos antes de la limpieza.


8. REDISEÑO DE PANTALLA DE ENTREGA
------------------------------------

  La página de detalle de actividad (/mod/activities/event/N)
  fue rediseñada completamente:

  Para el operador:
    - Si está pendiente: formulario limpio para subir evidencia.
    - Si ya completó: tarjeta de éxito, sin opción de resubir.

  Para jefe y admin:
    - Muestra quién completó la actividad y cuándo.
    - Muestra las observaciones del operador.
    - Botón directo para ver o descargar el archivo adjunto.
    - Si aún no hay entrega: mensaje "Sin entrega" claro.

  Se eliminó la etiqueta "Evidencia (solo lectura)" que
  generaba confusión. El diseño es ahora más formal y directo.
